# Кэширование

---

## 1. Базовый паттерн

Все кэширующие сценарии Bidding и Delivery — **Cache Aside** поверх Redis Cluster. Кэш закрывает главное узкое место AS-IS: тяжёлые чтения Bidding из Campaign PostgreSQL на каждом bid request, валящие hot path.

---

## 2. Что кэшируем

### 2.1. Hot path аукциона

| Ключ | Что хранит | Кто читает | Кто инвалидирует |
|---|---|---|---|
| `campaign:{campaign_id}` | active flag, `bid_ceiling_kop`, профиль таргетинга, идентификаторы креативов | Bidding | CDC из Campaign DB |
| `budget:{campaign_id}` | `remaining_kop`, `daily_cap_kop`, состояние pacing | Bidding | CDC + события impression из Kafka |
| `targeting:segment:{segment_id}` | множество user_ids в сегменте | Bidding | batch-job из Analytics |
| `freqcap:{user_id}:{campaign_id}` | счётчик показов в окне | Bidding | TTL + события impression из Kafka |
| `creative:{creative_id}` | разметка / URL / размеры креатива | Delivery Service | CDC из Campaign DB |

### 2.2. Сессии и rate limiting

| Ключ | Что хранит | Кто использует |
|---|---|---|
| `session:{token_hash}` | сессии пользователей кабинета | Campaign Service |
| `ratelimit:dsp:{dsp_id}` | sorted set с timestamps запросов | API Gateway |
| `ratelimit:ip:{ip}` | счётчик с TTL | API Gateway |
| `dedup:{event_id}` | флаг обработки события для Idempotent Consumer | Financial Service, Bidding Service (consumer-роль) |

### 2.3. Что НЕ кэшируем

- **Финансовые балансы.** Источник правды — Finance PostgreSQL primary; работать на stale-данных в денежных операциях недопустимо.
- **Сами bid responses.** Каждый bid request уникален, кэшировать ответы бессмысленно.
- **Креативы целиком (бинарные файлы).** Они в Object Storage; в Redis — только метаданные и URL.
- **Сырые impression/click события.** Они идут в Kafka, не в кэш.

## 3. Топология Redis Cluster

Redis Cluster развёрнут как managed-сервис (выбор провайдера зависит от текущего облака AdScale: AWS ElastiCache, Yandex Managed Service for Redis или аналог). Размещение — **в том же регионе и тех же зонах доступности, что и Bidding Service**, чтобы network latency между ними оставалась минимальной.

На 12-мес фазе при переходе к multi-region — отдельный Redis Cluster в каждом регионе с async-репликацией критичных ключей (см. [scaling.md §2.3](./scaling.md)).

**Внутри кластера:**

- Каждый шард: master + replica (failover внутри шарда, дополнительные чтения с реплики).
- Шардирование через **hash slot** от tag-части ключа в фигурных скобках — гарантирует, что связанные ключи попадают в один слот (`campaign:{12345}:profile` и `creative:{12345}:main` → один шард, можно использовать MGET одним батчем).
- Конкретное число шардов, persistence-настройки и eviction policy задаются DevOps-командой по результатам нагрузочного тестирования.

## 4. TTL стратегии

TTL выбирается под три характеристики данных: **частота изменений**, **критичность актуальности**, **объём изменений**. Главное правило: чем критичнее актуальность, тем короче TTL. Слишком короткий — обнуляет смысл кэша; слишком длинный — порождает stale data.

| Категория | Порядок TTL | Примеры в AdScale | Почему такой |
|---|---|---|---|
| **Короткий** | секунды–минуты | бюджеты кампаний, rate limit counters | бюджет меняется быстро под высоким трафиком; перерасход = финансовый риск |
| **Средний** | минуты–часы | кампании, креативы | меняются нечасто (правки рекламодателя); допустимо жить на устаревших данных в этом окне |
| **Длинный** | часы–сутки | сегменты таргетинга, счётчики frequency capping, сессии | сегменты пересчитываются raz в час batch-job; freq capping — счётчик показов за сутки |

Конкретные значения TTL подбираются командой на этапе настройки, исходя из реального профиля изменений и нагрузочного тестирования.

**Защита от cache stampede** (когда популярный ключ истекает и тысячи параллельных запросов идут в БД одновременно) — реализуется через single-flight locking или probabilistic early expiration на стороне клиента кэша. Деталь имплементации, не архитектурное решение.

## 5. Инвалидация

В AdScale применяется **гибрид трёх стратегий**.

### 5.1. По событию (event-driven через CDC)

Используется для критичных по актуальности данных: кампаний, бюджетов, креативов.

```text
Campaign Service ──UPDATE campaign──▶ Campaign PostgreSQL
                                            │
                                       Debezium (CDC)
                                            │
                                            ▼
                                  Kafka: cdc.campaign.v1
                                            │
                                            ▼
                                  Bidding consumer ──DEL campaign:{id}──▶ Redis
```

Bidding-consumer подписан на CDC-топики и удаляет соответствующие ключи в Redis при любом изменении. Следующий cache miss подтянет свежие данные.

Преимущество: точная актуальность сразу после изменения. Ограничение: задержка пути Postgres → Debezium → Kafka → consumer (несколько секунд) — приемлемо для нашего профиля.

### 5.2. По TTL

Используется для данных, где допустима задержка обновления в окне TTL: справочники, segment-индексы, frequency capping.

Простая, не требует дополнительных сервисов. Подходит, когда CDC избыточен.

### 5.3. По версии

Используется для конкурентной инвалидации, когда несколько инстансов могут одновременно обновлять кэш:

```text
key: campaign:{id}:v{version}
```

При обновлении кампании publisher увеличивает `version`, новые запросы автоматически идут по новому ключу, старая версия истекает по TTL. Защита от race condition «invalidation between read and write». Применяется выборочно — для creative-метаданных и больших targeting-индексов.

### 5.4. Сводка

| Тип данных | Стратегия |
|---|---|
| Кампании, креативы | по событию (CDC) + TTL как fallback |
| Бюджеты | по событию (CDC + события impression) + короткий TTL |
| Frequency capping | по TTL + явный INCR |
| Targeting сегменты | по TTL + перезапись batch-job |
| Sessions, rate limit | по TTL |

## 6. Прогрев кэша (cache warming)

После рестарта пода Bidding или развёртывания нового шарда Redis кэш пустой — каждый запрос идёт в Campaign PostgreSQL. На production-нагрузке это мгновенно перегрузит БД. Решение — прогрев кэша перед началом обслуживания трафика.

**Когда запускается warming:**

- **При деплое или рестарте Bidding-пода** — как Kubernetes initContainer перед стартом основного контейнера. Под не принимает трафик через Service, пока warming не завершён.
- **При создании нового Redis-шарда** — отдельный one-shot job, заполняющий шард данными.
- **По расписанию перед ожидаемым пиком** — например, перед массовыми событиями, когда команда знает, что трафик вырастет.

**Что прогревать:**

- Активные кампании (`status = 'active'`).
- Их бюджеты и creative-метаданные.
- Топ-N сегментов таргетинга.

Frequency capping и dedup-ключи не прогреваются — они быстро восстанавливаются от событий.

**Как прогревать безопасно:**

Главное правило — прогрев не должен сам перегрузить Campaign PostgreSQL. Поэтому warming работает поэтапно: пагинированное чтение из БД батчами с rate-limit, запись в Redis через pipeline (один round-trip на батч), логирование прогресса и измерение cache hit ratio после старта пода.

**Связка с canary rollout:** новые поды Bidding не получают трафик до того, как readiness probe вернёт OK. Readiness учитывает завершение warming и приемлемый cache hit ratio на тестовом трафике. Это исключает ситуацию «новый под обслуживает реальный трафик с пустым кэшем».

## 7. Сводная таблица решений по кэшированию

| Категория данных | TTL | Инвалидация | Прогрев |
|---|---|---|---|
| Кампании | средний | CDC + TTL fallback | да |
| Бюджеты | короткий | CDC + события impression | да |
| Креативы (метаданные) | средний | CDC + версия | да, топ-N |
| Targeting сегменты | длинный | TTL + batch-job | да, топ-N |
| Frequency capping | длинный | TTL + INCR | нет |
| Sessions | длинный | TTL | нет |
| Rate limit | короткий | TTL | нет |
| Dedup (`event_id`) | длинный | TTL | нет |
