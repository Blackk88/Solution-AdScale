# ADR-001: Архитектура эволюционирует через Strangler Fig + модульный монолит

- **Статус:** Принято
- **Дата:** 2026-05-23

---

## Контекст

AdScale — монолитное приложение на C++/Go/Python/Node.js с единственной инстанцией PostgreSQL, без кэша, очередей и контейнеризации. Текущий P95 latency (110–120 ms) уже превышает SLA (100 ms), запас по производительности отсутствует.

В ближайшее время подключается крупный DSP-партнёр, что даёт +20 % трафика немедленно (≈ 18 000 RPS) и до 2× за год (≈ 50 000 RPS). Контракт предусматривает SLA ≤ 80 ms и штрафы за downtime/latency.

Подробный анализ текущего состояния — в [AS-IS.md](../AS-IS.md). Полный набор архитектурных драйверов и ограничений — в [Drivers.md](../Drivers.md).

**Ключевые архитектурные драйверы для этого решения:**

- Нагрузка: 18 000 RPS на 3-мес фазе, 50 000 RPS к 12 мес, P95 ≤ 80 ms
- Срок: 3 месяца на промежуточное решение (BC-01)
- Непрерывность бизнеса — никаких окон простоя (TC-02)
- Нельзя переписать систему полностью (TC-01)
- Команда: 35 человек, нет SRE / тех. писателя / BA (BC-03)
- Ограниченный бюджет (BC-02)

**Рассматриваются три альтернативы стратегии эволюции:**

1. Оставить монолит и масштабировать вертикально
2. Big-bang переписывание на микросервисы
3. Strangler Fig Pattern + модульный монолит как промежуточная форма

---

### Альтернатива 1: Оставить монолит и масштабировать вертикально

Улучшения внутри монолита: in-process кэш, увеличение размера инстансов, read-replica.

- ➕ Минимальные операционные изменения, не нужно растить штат.
- ➖ Невозможно независимо масштабировать компоненты с разными профилями нагрузки (Auction Engine vs Statistics).
- ➖ Не устраняет SPOF на БД.
- ➖ Один процесс → один риск регрессии при релизе.
- ➖ Потолок вертикального масштабирования не позволит достичь 50 000 RPS.
- ➖ Не открывается путь к multi-region.

**Отклонено:** не закрывает целевые показатели даже на 3-мес горизонте.

### Альтернатива 2: Big-bang переписывание на микросервисы

Полная заморозка фич, переписывание системы за 6–12 месяцев, миграция с одномоментным переключением.

- ➕ Чистая целевая архитектура без legacy.
- ➖ Нарушает BC-01 (3-мес срок).
- ➖ Нарушает TC-02 (одномоментное переключение данных не обеспечивает непрерывность бизнеса).
- ➖ Штат 35 человек без SRE не способен одновременно эксплуатировать старую и новую систему 6+ месяцев.
- ➖ Высокий риск «распределённого монолита» — границы сервисов проверяются только теоретически до их экстракции.

**Отклонено:** нереалистично по сроку, штату и риску.

### Альтернатива 3: Strangler Fig Pattern + модульный монолит

Поэтапная экстракция доменов из монолита за edge router / API Gateway, начиная с самого критичного (bidding). Для ещё не выделенных доменов поддерживается дисциплина модульного монолита (чёткие границы, чтобы будущая экстракция была дешёвой).

- ➕ Соответствует TC-01, TC-02, BC-01.
- ➕ Каждый шаг приносит измеримое улучшение и снижает зависимость от монолита.
- ➕ Откат каждой экстракции — переключение трафика через API Gateway (минуты).
- ➕ Снижает риск «распределённого монолита»: границы проверяются на практике до экстракции.
- ➕ Постепенный подход — основной паттерн миграции из монолита в распределённую архитектуру для команд с ограниченным штатом.
- ➖ Период двойной поддержки (legacy в монолите + новый сервис).
- ➖ Требует дисциплины контрактов и API-версионирования.

**Принято.**

---

## Решение

Принимаем **Strangler Fig Pattern** как основную стратегию эволюции, с применением **модульного монолита** для ещё не выделенных доменов.

Конкретно:

1. Перед монолитом ставится **API Gateway** (Envoy / Kong / NGINX) — единая точка маршрутизации для DSP-трафика и UI. Это даёт инструмент для постепенного переключения трафика на новые сервисы.
2. Выделение доменов из монолита идёт **по одному**, в порядке убывания критичности по SLA и операционной боли:
   1. **Bidding Service** (Ad Server + Auction Engine) — первым (см. [ADR-002](./ADR-002-bidding-service-first.md)).
   2. Statistics Service + брокер событий (см. [ADR-003](./ADR-003-event-streaming-technology.md)).
   3. Analytics Service → отдельный OLAP-store.
   4. Financial Service → отдельная БД с strong consistency.
   5. Campaign Service + Advertiser Dashboard — последними.
3. Каждая экстракция включает: контейнеризация, перенос за API Gateway, теневой трафик → canary → полное переключение трафика → удаление legacy кода.
4. Внутри монолита **до экстракции** соблюдаются домены: запрещены межмодульные обращения в обход API-слоя, БД-таблицы помечены владельцами.
5. Окно экстракции одного домена — **≤ 4 недель**, чтобы минимизировать период двойной поддержки.

## Последствия

### Положительные

- 3-мес фаза достижима силами текущей команды: только bidding + Kafka + Redis + read-replica + API Gateway.
- Каждая итерация снижает риск нарушения SLA DSP-партнёра.
- Откат любого шага возможен через переключение трафика в API Gateway без миграции данных.
- Создаётся базовая платформа (API Gateway, Kafka, Redis, observability), которую переиспользуют последующие сервисы.
- Команда учится эксплуатировать распределённую систему постепенно, без шока.

### Отрицательные / Trade-offs

- Период двойной поддержки legacy в монолите и нового сервиса (≈ 4 недели на домен). Митигируется feature toggles и коротким окном.
- Контракты между сервисами и монолитом должны быть стабильны и версионируемы. Митигируется через OpenAPI/Protobuf в репозитории и контрактные тесты.
- Требуется усиление наблюдаемости с первого шага (без SRE — за счёт автоматизации, см. [Drivers.md](../Drivers.md)).

### Риски

- Декомпозиция застревает на полпути — монолит уже не цельный, но и не разобран, получаем «distributed monolith». Митигация: фиксированные milestone поквартально, owner per домен, ежемесячный архитектурный обзор.
- Неверно проведённые границы доменов приводят к дорогой переделке контрактов после экстракции. Митигация: event storming и валидация границ через модульный монолит до выноса, Protobuf-контракты в репозитории.
- Скрытая связанность модулей (общие таблицы, неявные допущения) обнаруживается только в момент экстракции и срывает сроки фазы. Митигация: аудит зависимостей модуля заранее, буфер 25 % на фазу.
- Регресс SLA в окне переключения между legacy и новым сервисом → штрафы от DSP-партнёра. Митигация: теневой трафик до переключения, canary 1 % → 10 % → 50 % → 100 % с автоматическим откатом по SLI.
- Расхождение данных в период двойной записи (dual-write), критично для финансов. Митигация: CDC как single source of truth, идемпотентность операций, reconciliation jobs, постепенное переключение reads → writes.
- Расхождение контрактов между сервисами без общего владения → интеграционные сбои. Митигация: контракт-ориентированная разработка, контрактные тесты в CI, semver и политика устаревания.
- Дефицит SRE-экспертизы → инциденты в продакшне новых сервисов и более длительные простои. Митигация: managed-сервисы (Kafka/Redis/Postgres), инструкции по эксплуатации, дежурство по ротации, инвестиция в observability с первого дня.
- Бюджетный перерасход из-за параллельной инфраструктуры (старая + новая). Митигация: отключение старых инстансов сразу после переключения трафика.
