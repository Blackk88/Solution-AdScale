## 2. Атрибуты качества

Категории по ISO/IEC 25010. Приоритет показывает, что определяет архитектуру: **P0** — критично, нарушение = срыв SLA или потеря клиентов; **P1** — обязательно на 12-мес горизонте.

### Performance — P0

| Метрика | 3 мес | 12 мес |
|---|---|---|
| Throughput аукциона | ≥ 18 000 RPS | ≥ 50 000 RPS |
| P95 bid response* | ≤ 80 ms | ≤ 80 ms |

\* Целевое значение определяется контрактным требованием DSP-партнёра — самым жёстким из применимых SLA.

### Scalability — P0

Горизонтальное масштабирование hot path без изменения кода; независимое масштабирование Bidding, Statistics, Analytics, Financial (разные профили нагрузки); линейный рост до 50 000 RPS; multi-region на 12-мес горизонте.

### Availability — P0

| Горизонт | SLA доступности |
|---|---|
| 3 мес | без регресса от текущего |
| 12 мес | 99.9 % (общесистемный) |

На 3-мес фазе ключевое требование к bidding-контуру — отсутствие регресса по сравнению с текущим состоянием.

### Reliability — P0

Graceful degradation Bidding при частичных отказах; идемпотентность финансовых операций; гарантированная доставка событий impressions/clicks; отсутствие SPOF на критическом пути. Конкретные паттерны реализации — в [adr/](./adr/) и [Task2/reliability.md](../Task2/reliability.md).

### Data integrity — P0 для финансов, P1 для остального

Финансы — strong consistency, ACID. Bidding — eventual consistency допустима (бюджеты могут проскальзывать в окне TTL). События — at-least-once с дедупликацией.

### Observability — P1

Метрики per-service (RED + ресурсные); distributed tracing запросов аукциона; структурированные логи с correlation ID; алерты на SLI/SLO; дашборды.

Повышенный приоритет из-за отсутствия SRE в штате — компенсируется автоматизацией наблюдения.

### Maintainability — P1

Возможность независимого деплоя выделенных сервисов; постепенная декомпозиция без рефакторингов с остановкой системы; версионируемые API-контракты; ADR для архитектурных решений.

### Security — P1

TLS на внешних вызовах (HTTPS для DSP-партнёров, рекламодателей, платёжного шлюза); аутентификация DSP-партнёров. Конкретные механизмы — в [Task2/api-gateway.md](../Task2/api-gateway.md) и [TO-BE.md](./TO-BE.md).

### Deployability — P1

Автоматизация развёртывания и переход от ручного деплоя к контейнеризации и автоматическому CI/CD — необходимое условие для эволюции через Strangler Fig (частые релизы, быстрый откат). Конкретные паттерны и таргеты — в [adr/ADR-002](./adr/ADR-002-bidding-service-first.md) и [TO-BE.md](./TO-BE.md).

### Top-3 driving forces

Архитектуру определяют (в порядке приоритета):

1. **Performance + Scalability** — нарушение = срыв SLA → штрафы → потеря партнёрства.
2. **Availability + Reliability** — downtime = прямые финансовые штрафы и репутация.
3. **Data integrity для финансов** — потеря/задвоение списаний разрушает доверие рекламодателей.

Enabling forces (без них первые три не удержать при росте): observability, deployability, maintainability.

## 3. Технические ограничения

| ID | Ограничение | Следствие |
|---|---|---|
| TC-01 | Нельзя переписать систему полностью | постепенная миграция (Strangler Fig) |
| TC-02 | Должна обеспечиваться непрерывность бизнеса | нет окон простоя, миграции выполняются без остановки системы |
| TC-03 | Текущий стек C++/Go/Python/Node | новые сервисы — на том же стеке, что команда уже эксплуатирует |
| TC-04 | PostgreSQL как основное хранилище | стратегия read-replica/sharding/CDC, а не миграция СУБД |
| TC-05 | Один production-кластер | стратегия HA → multi-AZ → multi-region |
| TC-06 | Нет SRE в штате | максимальная автоматизация и managed-сервисы |
| TC-07 | Нет технического писателя | документация ведётся разработчиками в репозитории |
| TC-08 | Нет бизнес-аналитика | требования формирует архитектор + менеджмент |
| TC-09 | Внешние интерфейсы DSP — OpenRTB | RTB endpoint поддерживает OpenRTB |

## 4. Бизнес-ограничения

| ID | Ограничение | Следствие |
|---|---|---|
| BC-01 | Срок 3 месяца на промежуточное решение | фокус на 4 первоочередные интервенции (см. [AS-IS.md §9](./AS-IS.md)) |
| BC-02 | Ограниченный бюджет | приоритет open-source / managed решениям, без enterprise-лицензий |
| BC-03 | Штат 35 человек, нет SRE/тех. писателя/BA | команда не может вести 10+ микросервисов одновременно — поэтапное выделение |
| BC-04 | DSP-партнёр уже подтвердил интеграцию | срок жёсткий |
| BC-05 | Текущие клиенты должны быть удержаны | нельзя ронять SLA существующих рекламодателей при миграции |
| BC-06 | Финансовые штрафы за downtime/latency | reliability и performance — P0 |
