# ADR-002: Bidding Service выделяется из монолита первым

- **Статус:** Принято
- **Дата:** 2026-05-23

---

## Контекст

В [ADR-001](./ADR-001-evolution-strategy.md) принят Strangler Fig Pattern — поэтапная экстракция доменов из монолита. Возникает следующий вопрос: **какой домен выделить первым** в условиях 3-месячного окна и подключения DSP-партнёра.

Кандидатами на первую экстракцию являются домены, помеченные как узкие места в [AS-IS.md §4](../AS-IS.md): Bidding, Statistics, Analytics, Financial.

**Ключевые архитектурные драйверы для этого решения:**

- Главная угроза — срыв SLA нового DSP-партнёра (≤ 80 ms), штрафы за латентность и downtime (BC-06)
- Срок 3 месяца на промежуточное решение (BC-01)
- DSP-партнёр уже подтвердил интеграцию — срок жёсткий (BC-04)
- Команда 35 человек, нет SRE — параллельные экстракции невозможны (BC-03)
- Должна быть обеспечена непрерывность бизнеса для текущих клиентов (TC-02, BC-05)

**Рассматриваются четыре альтернативы для первой экстракции:**

1. Сначала вынести Statistics
2. Сначала вынести Analytics
3. Сначала вынести Financial
4. Bidding Service первым (Ad Server + Auction Engine)

---

### Альтернатива 1: Сначала вынести Statistics

- ➕ Снимает write-нагрузку с БД быстро (Kafka + lightweight consumer).
- ➕ Технически проще: stateless ingestion + брокер, минимум бизнес-логики.
- ➖ Не решает главную проблему — sync DB-вызовы в hot path Auction Engine остаются.
- ➖ P95 Bidding не улучшится → SLA партнёру не выполняется.
- ➖ Stats — это «причина», но не «жертва»: эффект на латентность хоть и есть, но опосредованный.

### Альтернатива 2: Сначала вынести Analytics

- ➕ Тяжёлые SQL уйдут с prod БД, аукцион получит передышку.
- ➕ Можно сделать за счёт read-replica + перенос BI-нагрузки.
- ➖ Это уже стандартный паттерн (read-replica), не требует «выделения сервиса».
- ➖ Не закрывает SLA по Bidding — критический путь по-прежнему синхронен с PostgreSQL.

### Альтернатива 3: Сначала вынести Financial

- ➕ Изолирует деньги от остального — снижает риск финансовых сбоев.
- ➖ Не на hot path. Не помогает с SLA партнёру.
- ➖ Самая сложная экстракция (strong consistency, аудит, идемпотентность) — рискованно делать первой.

### Альтернатива 4: Bidding Service первым (Ad Server + Auction Engine) — **выбрано**

- ➕ Прямо закрывает главный риск контракта с DSP-партнёром — SLA ≤ 80 ms.
- ➕ Bidding на hot path → каждая микрооптимизация (кэш, async-вывод stats, read-replica) даёт измеримый эффект.
- ➕ Stateless-сервис проще всего масштабировать горизонтально → линейный рост throughput.
- ➕ Открывает дорогу остальным экстракциям: после bidding появляется опыт API Gateway, кэша, контейнеризации.
- ➕ Окупаемость с первой недели: даже частичный вынос (с in-process кэшем и async-emit в Kafka) снимает 40–60 ms.
- ➖ Самая критичная по риску экстракция: ошибка в hot path = немедленный финансовый ущерб.
- ➖ Требует одновременного вывода нескольких компонентов базовой платформы: API Gateway, Redis, Kafka, observability.

---

## Решение

Первым из монолита выделяется **Bidding Service** как самостоятельный **Bounded Context** (DDD), объединяющий функции текущих Ad Server и Auction Engine. Это даёт самый прямой эффект на SLA партнёра и открывает базовую платформу для последующих экстракций.

Bounded Context Bidding Service определяет три границы:

- **Границу модели:** единый язык домена (bid request, bid response, auction, winner, no-bid), инварианты аукциона.
- **Границу данных:** собственный Redis-кэш для кампаний/бюджетов и read-replica PostgreSQL; никакой другой сервис не читает эти хранилища напрямую.
- **Границу деплоя:** независимый жизненный цикл, отдельный контейнер, независимый rollout через API Gateway.

Объём 3-мес фазы для Bidding Service:

1. **Контейнеризация** Ad Server + Auction Engine как единый сервис (один deployable).
2. **API Gateway** перед монолитом и новым сервисом — единая точка входа для DSP, маршрутизирует трафик между legacy и Bidding Service. Детали процедуры переключения — в п. 7.
3. **Redis cluster** (managed, например Elasticache / Yandex Managed Redis) — read-through кэш кампаний, ставок, бюджетов, таргетинга. Цель — снять 40–60 ms с критического пути за счёт устранения sync-запросов к Postgres.
4. **Kafka** между Bidding/Delivery и Statistics Service — асинхронный поток событий (см. [ADR-003](./ADR-003-event-streaming-technology.md)). Statistics Service в монолите остаётся consumer'ом, но теперь не на hot path.
5. **Read-replica PostgreSQL** для бэка аналитики и UI — устраняет конкуренцию с OLTP.
6. **Базовая observability**: метрики SLI (latency p95/p99, error rate, RPS, cache hit ratio), distributed tracing, алерты. Без этого нельзя подтверждать SLA партнёру.
7. **Процедура переключения трафика:** теневой трафик → canary 1 % → 10 % → 50 % → 100 % с автоматическим откатом по SLI. Окно постепенного наращивания — 2 недели.

Что **не** входит в 3-мес объём Bidding Service (отложено):

- Kubernetes — фаза 6 мес (на 3-мес фазе допустимо managed-VM с автоскейлинг-группой).
- Multi-region — фаза 12 мес.

## Последствия

### Положительные

- К концу 3 мес SLA партнёра (≤ 80 ms p95) выполнен → нет штрафов, контракт удержан.
- Throughput Bidding масштабируется независимо от Statistics / Financial / Analytics.
- Появляется базовая платформа (API Gateway, Redis, Kafka, observability), которую переиспользуют остальные сервисы — экономия времени на каждой следующей экстракции.
- Команда получает практический опыт операционного управления распределённой системой на одном, самом важном, домене.
- Откат через API Gateway — переключение трафика обратно на монолит без миграции данных.

### Отрицательные

- Высокий риск первой экстракции: ошибка в hot path = немедленный ущерб. Митигируется теневым трафиком + canary.
- Одновременный вывод нескольких компонентов базовой платформы (Gateway + Redis + Kafka) — сложно для команды без SRE. Митигируется выбором managed-сервисов и узкими зонами ответственности (один человек / один компонент).
- В монолите на 3-мес фазе остаются Delivery, Statistics, Financial, Analytics, Advertiser Dashboard — продолжается совместная эволюция кодовой базы вместе с новым Bidding Service. Митигируется чёткими API-контрактами и feature toggles.

### Риски

- Регресс SLA в окне постепенного наращивания canary. Митигация: автоматический откат при превышении p95 > 100 ms на 5 минут подряд, ручной аварийный переключатель в API Gateway.
- Несогласованность кэша Redis с PostgreSQL → Bidding работает на устаревших бюджетах и превышает их. Митигация: короткий TTL (≤ 30 с) + CDC-инвалидация по Kafka + max-overage threshold (например, +5 % к бюджету) с защитной отсечкой.
- Потеря событий impressions/clicks при включении Kafka между Delivery и Stats. Митигация: at-least-once gateway → дедупликация на потребителе, replication factor ≥ 3, acks=all, мониторинг lag.
- Деградация монолита в окне переключения: Bidding Service уже вытаскивается, но общая БД-нагрузка не падает мгновенно. Митигация: одновременное включение read-replica и кэша до переключения трафика.
- Эксплуатация Redis/Kafka без SRE → инциденты в продакшне. Митигация: только managed-сервисы на 3-мес фазе, дежурство по ротации, инструкции по эксплуатации на ключевые сценарии (split-brain, partition, replica lag).
- Контрактные различия между OpenRTB-вызовами от разных DSP. Митигация: API Gateway как унификатор (валидация и нормализация), versioned адаптеры в Bidding Service.
