# Текущая архитектура AdScale: диагностика

Анализ текущего состояния платформы. Покрывает четыре направления: узкие места, точки отказа, проблемы масштабирования, влияние на latency.

---

## 1. Контекст

AdScale — монолитное приложение с компонентами на C++/Go/Python/Node.js, работающее как один процесс. Единственная PostgreSQL обслуживает все домены — кампании, ставки, события, финансы, аналитику. Нет кэша, очередей, контейнеризации; деплой ручной, один production-кластер.

Система уже работает почти на пределе текущих возможностей: запас по производительности отсутствует, в часы пик наблюдаются деградации.

| Параметр | Значение | Требование |
|---|---|---|
| Пиковый RPS | 15 000 | — |
| P95 latency | 110–120 ms | ≤ 100 ms |
| Деградации в часы пик | периодические | не допускаются |
| Запас по производительности | отсутствует | требуется |

Подключение крупной DSP-площадки добавит +20 % трафика немедленно (≈ 18 000 RPS) и до 2× за год (≈ 50 000 RPS). Партнёр требует p95 ≤ 80 ms и штрафует за деградацию.

## 2. Карта компонентов

| Компонент | Язык | На hot path аукциона | Роль |
|---|---|---|---|
| Ad Server | C++/Go | да | приём bid request, подбор кандидатов |
| Auction Engine | C++ | **критично** | взвешивание ставок, выбор победителя |
| Delivery Service | Node.js | да | формирование HTML/JS-разметки |
| Statistics Service | Python | нет (write-нагрузка) | запись кликов/показов |
| Financial Service | Go | нет | списания, балансы |
| Analytics Service | Python | нет (read-нагрузка) | SQL по prod-БД |
| Advertiser Dashboard | Node.js | нет | UI рекламодателя |
| PostgreSQL | — | да (через Auction, Ad Server) | единое хранилище |

Все компоненты делят один процесс и одну БД.

## 3. Критический и некритичный потоки

Критичные и некритичные потоки в AdScale делят единый рантайм и единую PostgreSQL. Чтобы зафиксировать, что именно надо разделить на 3-мес фазе, явно выделяем путь bid response и потоки, в нём не участвующие.

### 3.1. Критический путь (bid response ≤ 80 ms)

Последовательность, через которую проходит каждый bid request от DSP до возврата ответа. Срыв SLA на любом шаге = штраф партнёра.

```text
DSP-партнёр
    ↓  HTTPS / OpenRTB
Ad Server ─────read───── PostgreSQL   (чтение данных для таргетинга)
    ↓  Internal call
Auction Engine ──read──── PostgreSQL  (ставки и кампании)
    ↓  Internal call
Delivery Service
    ↓  Internal call
Statistics Service ─write─ PostgreSQL (запись показа/клика)
    ↓
bid response → DSP-партнёр
```

Компоненты на критическом пути:

| Компонент | Роль на пути | Аннотация |
|---|---|---|
| Ad Server | приём bid request, чтение таргетинга из БД | `<<Bottleneck>>` |
| Auction Engine | sync-чтение БД, взвешивание, выбор победителя | `<<Critical Path>>` |
| Delivery Service | формирование разметки победителя | — |
| Statistics Service | sync-запись события в БД | `<<Bottleneck>>` (write) |
| PostgreSQL | единая БД для всех чтений и записи на пути | `<<Single Point of Failure>>` |

### 3.2. Некритичные потоки

Не нужны для возврата bid response. Допустима задержка секунд-минут.

| Поток | Цепочка |
|---|---|
| Управление кампаниями и бюджетами | Рекламодатель → Advertiser Dashboard → PostgreSQL |
| Пополнение бюджета | Рекламодатель → Advertiser Dashboard → Financial Service → Платёжный шлюз |
| Финансовые транзакции (списания, балансы) | Financial Service → PostgreSQL |
| Аналитические запросы и отчёты | Advertiser Dashboard → Analytics Service → PostgreSQL |

Компоненты вне критического пути: Financial Service, Analytics Service, Advertiser Dashboard.

### 3.3. Где разделения нет

Несмотря на разный приоритет по latency, критичные и некритичные потоки конкурируют за общие ресурсы:

- **Единая PostgreSQL** — Auction Engine (read для аукциона) делит инстанс с Financial (write транзакций) и Analytics (heavy SELECT). Конкуренция за блокировки и IOPS.
- **Sync-запись событий с критпути** — Delivery → Statistics → PostgreSQL выполняется синхронно внутри окна bid response, write-нагрузка от показов/кликов влияет на latency аукциона.
- **Аналитика в production OLTP** — длительные чтения блокируют те же таблицы, на которых работает Auction Engine.
- **Один монолитный процесс** — все компоненты в одном runtime; релиз или баг в любом из них может уронить bidding.

Разделение этих потоков — одно из ключевых требований 3-мес фазы и определяющая мотивация для TO-BE-архитектуры.

## 4. Узкие места

### 4.1. Auction Engine — критический bottleneck

- **Синхронные обращения к PostgreSQL** в hot path при бюджете latency ≤ 100 ms.
- На каждом запросе — ожидание disk I/O и lock wait в БД, которая параллельно обслуживает запись Stats и тяжёлые SQL Analytics.
- Нет кэша горячих сущностей (кампании, ставки, бюджеты), нет prefetch.
- Любое замедление БД немедленно превращается в превышение SLA.

### 4.2. Statistics Service — write-bottleneck

- Запись показов и кликов идёт напрямую в общую PostgreSQL — буферизация отсутствует, поток событий создаёт высокую write-нагрузку и конкурирует с transactional-запросами финансов.
- При росте трафика поток событий быстро упрётся в потолок общей PostgreSQL.

### 4.3. Analytics Service — read-bottleneck

- Тяжёлые аналитические SQL идут в ту же production-БД, где работают аукцион и финансы — отдельной реплики или OLAP-хранилища нет.
- Длительные чтения вызывают блокировки и рост latency на критическом пути аукциона.

### 4.4. Ad Server — частичный bottleneck

- Читает таргетинг-данные напрямую из БД. Те же ограничения, что у Auction Engine.
- Логика подбора живёт в том же процессе — деплой одной правки требует выкатки всего монолита.

## 5. Точки отказа (SPOF)

| SPOF | Последствие | Текущая защита |
|---|---|---|
| Единственная инстанция PostgreSQL | полная остановка всех потоков (аукцион, биллинг, аналитика, UI) | отсутствует HA / failover |
| Один production-кластер | региональный сбой = глобальный отказ | нет multi-region |
| Монолитный процесс | падение/утечка в любом модуле = деградация остальных | нет изоляции (общий heap, общий пул соединений) |
| Synchronous coupling Auction → DB | замедление БД → срыв SLA аукциона | нет circuit breaker / fallback |
| Отсутствие очередей перед Stats | пик трафика = потеря событий или обратное давление в hot path | нет буферизации |

## 6. Проблемы масштабирования

1. **Невозможно масштабировать компоненты независимо.** Auction Engine упирается в CPU, Statistics — в write-IOPS, Analytics — в память и I/O. Монолит масштабируется только целиком, что неэффективно для разных профилей нагрузки.
2. **OLTP и OLAP в одной БД.** Аналитика и аукцион конкурируют за buffer pool и disk I/O. Любой тяжёлый отчёт = риск SLA.
3. **Нет кэша.** Каждый bid request обращается к диску за статичными и полустатичными сущностями.
4. **Нет асинхронной обработки.** Stats и Financial пишут синхронно, связывая критический путь с долгими операциями.
5. **Жёсткая связность модулей.** Изменения в одном модуле требуют регрессии всей системы.
6. **Нет горизонтального шардирования.** PostgreSQL вертикально упирается в потолок одной машины.
7. **Нет контейнеризации и оркестрации.** Невозможен автоскейлинг и быстрый rollout.

---

## 7. Влияние на latency

Бюджет — 100 ms (текущий) и 80 ms (после подключения нового DSP-партнёра). Текущий P95 — 110–120 ms, бюджет уже превышен.

Основные драйверы latency и их роль уже описаны в §4 (узкие места): синхронные обращения к БД из hot path аукциона, конкуренция за общую PostgreSQL со Stats и Analytics, отсутствие кэша на подборе кандидатов. Под ростом трафика +20 % все они деградируют нелинейно — задержки увеличиваются быстрее, чем сама нагрузка, и удержать целевой SLA (≤ 80 ms) в текущей архитектуре невозможно.

---

## 8. Прогноз деградации

| Сценарий | Trigger | Последствие |
|---|---|---|
| +20 % bid requests (≈ 18 000 RPS) | подключение DSP-партнёра | срыв SLA 80 ms, штрафы, риск отказа партнёра от интеграции |
| Пик в часы максимальной активности | сезонность, event-кампании | блокировки в БД, рост P95 до 200+ ms, downtime финансового сервиса |
| Сбой PostgreSQL | hardware / disk / replication | полный downtime платформы |
| Heavy analytics запрос рекламодателя | действия в UI | каскадная деградация аукциона для всех клиентов |
| 2× рост к концу года (≈ 50 000 RPS) | органический + DSP | архитектура физически не выдерживает: одна БД, один кластер, монолит |
